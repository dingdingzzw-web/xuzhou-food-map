## 今日已完成

### 1. 云端部署跑通
已完成：
- 将前端项目部署到 Vercel
- 修正项目部署根目录为 `web`

### 2. 地图能力从展示骨架升级为可缩放地图
已完成：
- 放弃继续把行政区 GeoJSON + SVG 强行扩展成街道级地图
- 评估后改为：**MapLibre 作为主地图底层**
- 首页主地图已支持：
  - 拖拽
  - 缩放
  - 店铺 marker 展示
  - marker 点击联动右侧详情

相关提交：
- `7570969` feat: add zoomable maplibre map

结论：
- GeoJSON 行政区底图适合概览展示，不适合承载街道级缩放体验
- 真正想做“能逛”的地图，还是要引入可缩放底图能力

---

### 3. 上传店铺接入地图选点
已完成：
- 上传面板支持接收地图选点坐标
- 点击左侧地图可给上传表单带入经纬度
- 后续又修正为：
  - 默认浏览模式下不自动抢选点
  - 需要手动开启“地图选点模式”
  - 选出的蓝点支持拖动微调

相关提交：
- `e1baf0d` feat: add map picker for shop uploads
- `2dc04fe` fix: improve map picking and geocode feedback

结论：
- 选点功能必须是“显式模式”，不能让地图一碰就写坐标
- 对普通用户来说，自动定位优先，地图选点应该只作为修正手段

---

### 4. 地址自动定位打通
已完成：
- 上传表单新增“自动定位”按钮
- 初版尝试直接使用旧高德 Web JS key 调 REST geocode 接口，失败
- 通过 Network 抓包定位到错误：
  - `USERKEY_PLAT_NOMATCH`
  - `INVALID_USER_KEY`
- 最终确认原因：
  - 旧 key 类型不对
  - 需要单独创建 **高德 Web服务 key** 用于 geocode REST API
- 更换正确 key 后，地址自动定位已可正常工作

当前上传流程已变成：
1. 输入店名
2. 输入地址
3. 点击自动定位
4. 地图跳转到对应位置
5. 如有偏差，再开启地图选点模式微调
6. 提交上传

相关提交：
- `1bfe27e` feat: auto locate shop address with amap

结论：
- MapLibre 做底图，高德只做 geocode，是当前最适合这个项目的组合
- 高德 key 需要按调用场景区分平台类型，不能混用

---

### 5. Supabase 从零接入真实环境
已完成：
- 用户新注册 Supabase 项目
- 已确认并配置：
  - `NEXT_PUBLIC_SUPABASE_URL`
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- 已执行 `supabase_schema_v1.sql`，建表成功
- 原始 RLS SQL 在当前环境下出现兼容性问题：
  - `create policy if not exists` 语法报错
- 已衍生兼容版本：
  - `supabase_rls_storage_v2.sql`
- 后续又因 `storage.objects` 权限不足报错：
  - `must be owner of table objects`
- 最终拆出仅针对业务表的版本：
  - `supabase_rls_storage_v3.sql`
- `v3` 已成功执行

当前数据库层状态：
- `shops`
- `shop_images`
- `shop_votes`

已具备基本 RLS 规则，足够支撑当前 MVP 的店铺读写闭环。

结论：
- 当前阶段先别纠缠 Storage policy
- 先把“店铺读写 + 自动定位 + 前端地图交互”跑通更重要
- 图片上传可放到下一阶段再补
